Front ends & back ends

There's no single right stack — there's the right stack for the job. Everything below runs in one of my shipped projects; the trade-offs are the ones I actually hit, not the ones in the docs.

Front ends

From zero-dependency vanilla JS to a typed Next.js app — matched to how interactive and data-heavy the UI is.

Stack Best for Pros Cons Used in
Vanilla JavaScript no framework at all Lean, long-lived UIs Games, editors, app shells — anywhere framework weight buys nothing. Zero dependencies, zero build Nothing to update or break; runs natively in every browser. You hand-roll the plumbing Routing, state and components are yours to write and maintain. Games · Learning Lab · Golf Coach Including a hand-built SVG/canvas mind-map editor with drag, connect and auto-layout.
React + Vite interactive SPAs Rich, chart-heavy interfaces Component model pays off once the UI has real state to manage. Fast builds, huge ecosystem Recharts, Tailwind and PWA tooling drop straight in; instant dev reload. A build step and a dependency tree More moving parts to keep current than plain JS. RunTracker · Family Manager Interactive charts in one; an installable offline PWA in the other.
Next.js + TypeScript + Tailwind typed, server-rendered apps Data-heavy apps that need SSR Server rendering for fast first paint and SEO; types keep a big data UI honest. SSR + typed components App Router, generated DB types end-to-end, utility CSS for speed. Heaviest stack here Framework churn and build complexity; overkill for simple pages. Schoolwise Maps, filters and multi-year charts over a large schools dataset.
Django templates server-rendered HTML Transactional sites Shops, forms, admin — pages that submit and reload beat a SPA here. Simple, robust, SEO-friendly One codebase for front and back; no API layer to design or version. Limited client-side interactivity Rich in-page behaviour still needs sprinkled JS on top. Retail Platform Web shop, till and kiosk all rendered server-side.

Back ends

From batteries-included Django to Python at the edge — and knowing when the right back end is none at all.

Stack Best for Pros Cons Used in
Django / Python batteries-included framework Full business apps Auth, admin, ORM and payments integration without assembling parts. Everything included and mature django-allauth for sign-in, admin for back office, migrations for the schema. Needs a real server Long-running process to host, patch and monitor — not serverless. Retail Platform Django 6 on Python 3.13, on an always-free GCP VM.
Python Cloudflare Workers Python at the edge APIs with nothing to operate Global edge deploys with storage (D1, KV, Durable Objects) alongside. No server, instant deploys Hand-rolled routing keeps the bundle lean and cold starts fast. Worker runtime, not CPython-anywhere No filesystem; not every library runs; CPU time per request is capped. Learning Lab · RunTracker Including real-time multiplayer over WebSockets and Durable Objects.
Zero-dependency Node plain http, no Express Small self-hosted tools When the whole API is a handful of routes, a framework is overhead. Nothing to install or audit No node_modules, no supply-chain surface, trivially portable. Everything is manual Routing, parsing and errors are hand-written; doesn't scale to big APIs. Golf Coach Serves the mind-map editor and the knowledge-base API.
Supabase Edge Functions + Python pipeline generated API + portable ingest Data products over Postgres PostgREST generates the read API; custom code only where it earns its keep. Most of the back end is free Auto REST off the schema; the Python ingest pipeline runs anywhere. Shaped by PostgREST Complex queries lean on RPC functions; some platform lock-in. Schoolwise Yearly government datasets ingested by Python, served by PostgREST.
No back end everything in the browser Single-user & static apps If nothing must be shared or secret, the browser is the whole platform. Nothing to run, ever No server costs, no ops, no downtime; data stays on the user's device. No sync, no secrets Cross-device sharing and private keys are off the table by design. Games · Family Manager Static files on Pages; an offline-first PWA on Vercel.
